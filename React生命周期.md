# React生命周期
### [前言]
一个优秀的框架需有一套完善的生命周期，而了解生命周期有利于对框架进行深度解读。如果能够熟练的掌握生命周期，那么在进行业务建模以及实现的时候会更加方便。React的生命周期是非常清晰简单的，在这里我将使用ES6的语法来分析React的生命周期，React生命周期分三个阶段，其实在我看来，一共有四个阶段，额外的那个阶段就是实例化的调用constructor方法，这个是对象在被实例化都会调用，不过我觉得它也可以成为一个阶段。每个完整生命周期代表着一个业务，接下来就用阅读这样一个业务来分析生命周期中每个阶段每个方法。
###[分析]
在React的世界里有两个最重要的东西：**props**和**state**，我们可以将props理解成外因，state理解成内因，那么我们来分析下阅读的内因和外因，首先外因里面应有时间因素，环境因素等等，而内因主要是对书本的理解和知识点的掌握。有了这两个前提我们才能更好的在各个阶段做阐述。
###[STAGE]实例化

```
class ReadComponent extends React.Component{
    state = {}
    constructor(props){
        super(props);
        // this.state = {}
    }
}
```
在这个阶段能干什么呢，很多人会在这里面进行对状态数据进行初始化，我是不推荐这么做，而是直接在类的内部定义成属性。首先，在组件实例化的时候，作为类的属性应该已经存在，而不是在实例化的时候才去创建；然后，组件实例化的时候应该保证状态应该是空的。就像你准备阅读的时候，你对这书的状态应该以空的状态已经存在了。在实例化阶段我建议去做一些固定类数据的初始化，而固定数据是不需要存储在state里面的，里面阅读前你要准备一本书，一本笔记本和一支笔，这些会贯穿整个阅读过程，也不会因为你的阅读体会发生变化而变化，所以这些数据可以放在实例化阶段来完成，如果这部分数据不受props影响，最好不要去重写constructor方法。
###[STAGE-MOUNT]
这个阶段完成的将抽象节点写入到文档中变成实际节点。主要由componentWillMount和componentDidMount完成。顾名思义，这个分别触发于挂载前和挂载后。这是最常用的，也是我认为React最好的两个方法。

```
class ReadComponent extends React.Component{
    componentWillMount(){}
}
```
对state的数据初始化我建议在这里完成，为什么呢？就像你在看书前，你关注过的书评和对书本身的期待在这个时候触发的。

```
class ReadComponent extends React.Component{
    componentDidMount(){}
}
```
当一本书第一次看完的时候，你需要写一篇读后感，或者记一些笔记。在读完之后，如果你认为这本书是垃圾，可能直接把书给扔了，这时候直接进入销毁阶段。有时候你会在读完之后又有所感悟，但是又不是那么深刻（*要是你认为你已经深刻理解了，就可以把书收藏起来了，这时候又进入销毁节点*），这时候你的认知状态发生了，你还想继续读下去，所以就回去重新看，这样我们就进入了更新阶段。在当前阶段，业务上一般把第一次数据请求放在这里，如果我们把数据请求放在componentWillMount里面，会有一个问题是如果请求速度快于render的速度，那么页面不会渲染，并且会抛出警告。但是放在这里请求又有一个问题是刚开始的首屏数据是空的或者页面是空的，所以在这里要做好友好的过度。还有一些针对dom节点的事件和一次性操作都可以放在这里。
###[STAGE-UPDATE]
这个阶段是非常重要的，是你能不能将一本书完全参透的重要阶段，这个阶段由于节点已经渲染，所以这部分主要是不断更新节点，而React在更新之前会去计算更新哪些节点，这里有React最具争议的diff算法，在数据量小的情况，diff不会出现性能瓶颈，而当数据量很大，设计到的节点过多，这个算法其实并不理想，这时候就需要我们人为的优化了。

```
class ReadComponent extends React.Component{
    componentWillReceiveProps(props){}
}
```
当外界的环境变化了，大家对你看的书的评价发生了变化，或者出现了新的阅读技巧或者更好的工具。在外界环境变化的时候componentWillReceiveProps会被触发，告诉你外界的变化是哪些，而承载变化信息的就是我们的props，在这里可以根据外界变化调整内在状态。在实际业务场景中，我就遇到一个非常好的例子，在react-router里面设置了一个详情页，假设路由是"/user/:id",这个页面就是查看用户信息，但是用户信息里面有链接查看其它人的详情，这时候点击链接，页面是不会销毁的，这时候这个方法就非常有用，当ID发生变化的时候重新请求数据，并更新state。

```
class ReadComponent extends React.Component{
    shouldComponentUpdate(props, state){
        return {boolean};
    }
}
```
你在阅读之前给自己设置了一系列的规则告诉自己那些情况下才能继续阅读，这时候shouldComponentUpdate就会帮你大忙。在前面我们说过diff算法在数据复杂度很高的情况下会出现性能问题，那么这个问题基本就是用来帮助你解决性能问题的利器，在这里只要返回false，更新就不会进行。当然也有例外，有一个霸气外露的方法可以直接跳过他，这个方法叫forceUpdate(),看名字就知道很霸气有木有。一句话，只要我出马，无论如何都要给我更新。

```
class ReadComponent extends React.Component{
    componentWillUpdate(){}
}
```
这个其实没啥好说的，跟componentWillMount差不多，就是你再次阅读前需要做的准备事宜。反正我用到的很少。

```
class ReadComponent extends React.Component{
    componentDidUpdate(){}
}
```
这里我们得恭喜你，你又读完一遍了，但是不幸的是你不能在这里更新你的认知状态，因为你的认知状态在这之前已经完成了。还有一种可能就是我举的例子不恰当。这个时候跟componentDidMount差不多，都是页面渲染完后的处理，这里一般不会对节点进行事件处理，更多的是样式处理，例如你有一个页面要不断的输出服务端的日志，而且每次刷新后都要滚到最后面，这里就非常合适。
###[STAGE-DESTROY]
在这里，你的阅读已经进入最后部分，要么是你不喜欢直接扔书，要么就是读完了准备把书和工具都收拾好了。

```
class ReadComponent extends React.Component{
    componentWillUnmount(){}
}
```
在真实节点即将被移除之前，我们需要把注册的乱七八糟的事件全部销毁，还有什么GC无法回收的数据也要在这里销毁。
### 杂谈
我感觉阅读这个例子不能很好体现生命周期与状态、外因直接的关系，但是思维有限我也想不出来什么很好的例子。最后来一个小点，在React里面非常重要的key，这是一个让我们又爱又恨的属性，不过也是非常好用的属性。如果你对一个组件恨之入骨，请把时间戳复制给这个组件的key。
### (0\^_\^0)结束语
至此我们完成了一个完整的生命周期，也完成了一个完整的业务。理解业务也就是理解生命周期，通过对生命的周期了解，不断去拆分我们的业务，完成最优的实现。当然在任何地方你可以做任何实现，我们只希望在合理的地方做合适的事情，哪怕他可能会复制点，至少不会看上去像野路子，而在大型项目，尤其是在工程化项目里面，最需要注意这些，这样可以让代码跟利于阅读和跟踪。
